## 🔺 Обзор
Nekhebet — модульная система создания, подписания, проверки  
и хранения **криптографически верифицируемых событий**.

Проект состоит из независимых компонентов:
* **Core** — криптографическое ядро
* **Store** — верифицируемое append-only хранилище
* **Adapters** (ingest) — адаптеры приёма событий

Все компоненты следуют zero-trust модели и фиксированным инвариантам безопасности.

## 🔺 Назначение
Система решает задачу:
> Доказуемо установить аутентичность, целостность и уникальность событий  
> независимо от источника и среды выполнения.

Nekhebet предназначен для:
* аудита и комплаенса
* event-driven архитектур
* приёма данных из недоверенных источников
* форензики и воспроизводимых пайплайнов

## 🔺 Ключевой инвариант
> **Событие существует только если его подпись и структура валидны.**

Если событие принято:
* оно подписано Ed25519 по каноническому представлению
* защищено от replay-атак
* может быть проверено детерминированно в любой среде

## 🔺 Архитектурная модель
```
External sources
        ↓
Adapters (ingest)
        ↓
Core → signing & verification
        ↓
Store → append-only persistence
        ↓
Consumers (audit, analytics, visualization)
```

Каждый компонент:
* решает одну задачу
* не зависит от транспорта
* сохраняет глобальные инварианты

## 🔺 Компоненты
### Core
Самодостаточное криптографическое ядро.

Функции:
* канонизация по RFC 8785 (JCS)
* подпись Ed25519
* строгая верификация
* защита от replay
* применение политик

<sub>**Реализация:** чистый Python, без внешних зависимостей для криптографии.</sub>

### Store
Верифицируемое постоянное хранилище.

Функции:
* append-only модель
* повторная верификация при записи
* индексация и поиск
* уведомления о новых событиях

<sub>**Реализация (reference):** PostgreSQL + blob storage.</sub>

### Adapters (Ingest)
Адаптеры приёма из внешних источников.

Функции:
* нормализация данных
* интеграция с внешними системами
* передача в Core для подписи/проверки

<sub>**Реализация (reference):** асинхронные адаптеры (включая Telegram-источники).</sub>

## 🔺 Формат события (SignedEnvelope)
```json
{
  "header": { "key_id", "nonce", "issued_at", ... },
  "payload": { /* arbitrary domain data */ },
  "signature": "Ed25519"
}
```
* Канонизация: строго по JCS
* Хэш: SHA-256
* Replay-защита: (key_id, nonce, issued_at)

## 🔺 Обработка
Допускается вторичная обработка после верификации:
* индексация
* агрегация
* экспорт
Эти операции не влияют на доверенную модель.

## 🔺 Модель состояния
* Core — stateless
* Store — единственный источник истины
* Adapters — best-effort приём

## 🔺 Устойчивость
Система спроектирована так, чтобы:
* сбои не нарушали целостность уже записанных событий
* компоненты могли перезапускаться независимо
* валидность не зависела от времени или среды

## 🔺 Инварианты
1. Фиксированный алгоритм подписи (Ed25519)
2. Детерминированная канонизация (JCS)
3. Защита от replay-атак
4. Полная верификация перед любой операцией
5. Zero-trust: источник не влияет на валидность
