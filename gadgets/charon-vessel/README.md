## 🔺 Обзор
Charon Vessel — демон ротации файлов для Linux и FreeBSD  
с **атомарными операциями** и строгой защитой от race-condition атак.

Проект реализован как **один самодостаточный бинарный файл**  
без внешних runtime-зависимостей, с минимальным доверенным кодом  
и явным отказом от любых небезопасных состояний.

## 🔺 Назначение
Демон решает задачу:
> Надёжно и предсказуемо управлять жизненным циклом файлов  
> в условиях потенциально враждебной среды.

Charon Vessel гарантирует:
* атомарное перемещение файлов в архив
* защиту от TOCTOU, symlink, hardlink и других атак
* строгую валидацию всех путей и операций
* отказ при малейшем подозрении на небезопасное состояние

## 🔺 Ключевой инвариант
> **Файл либо полностью свежий, либо полностью архивирован — третьего не дано.**

Если операция завершена успешно:
* файл находится в одном из строго определённых состояний
* все промежуточные состояния исключены
* операция может быть повторно проверена

## 🔺 Архитектурная модель
```
Incoming files → FRESH directory
                 ↓ (atomic rename when limit reached)
               ARCHIVE directory (timestamped)
                 ↓ (parallel copy on cross-device)
               Final archived state
```

Каждый шаг:
* использует только системные примитивы
* проверяет безопасность до выполнения
* не оставляет артефактов при отказе

## 🔺 Компоненты
### Основной демон
Единственный исполняемый процесс.

Функции:
* мониторинг FRESH-директории
* атомарная ротация при достижении лимита
* параллельное копирование при cross-device
* сбор внутренних метрик

<sub>**Реализация:** один C++17 исходник, компилируется в статический бинарник.</sub>

### Конфигурация
Управление только через переменные окружения.

Ключевые параметры:
* пути к директориям
* лимиты количества/размера
* параметры параллелизма
* уровни проверок безопасности

### Мониторинг
Команда `status` возвращает:
* текущее состояние
* счётчики операций
* внутренние метрики

## 🔺 Ограничения и допущения
* Директории должны существовать до запуска
* Имена с `..` отклоняются намеренно
* Требуются минимальные capabilities/root для безопасных операций
* FreeBSD: без io_uring и xattr

## 🔺 Жизненный цикл файла
```
Создание во FRESH
 → мониторинг количества
 → достижение MAX_FRESH
 → атомарный rename/link
 → (при необходимости) копирование
 → финальное состояние в ARCHIVE
```

## 🔺 Устойчивость
Демон спроектирован так, чтобы:
* сбои не оставляли файлов в промежуточном состоянии
* перезапуск восстанавливал корректное поведение
* отказы были явными и логируемыми

## 🔺 Инварианты
1. Никаких race-condition в операциях ротации
2. Никаких symlink/hardlink атак
3. Никаких частично выполненных операций
4. Полная валидация путей до любого действия
5. Отказ предпочтительнее небезопасного продолжения
