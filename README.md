## Назначение и область ответственности

**Nekhebet** — это инфраструктурная система для создания, подписи, верификации и хранения *криптографически защищённых событий* в виде подписанных конвертов (signed event envelopes).

Проект решает класс задач, связанных с:

* доказуемой целостностью данных,
* подтверждением источника события,
* защитой от повторного воспроизведения (replay),
* детерминированной сериализацией,
* безопасным пересечением границ доверия между системами.

Nekhebet не является прикладным продуктом. Это **базовый строительный слой**, на котором могут быть реализованы системы аудита, event-driven платформы, ingest-пайплайны, защищённые журналы событий и распределённые хранилища.

---

## Ключевая идея

Любое событие представляется как **SignedEnvelope** — неизменяемая структура, содержащая:

1. канонический заголовок,
2. полезную нагрузку (payload),
3. криптографическую подпись.

Подписывается **не сериализация**, а **каноническое представление данных**, что обеспечивает воспроизводимую и независимую верификацию в любой среде.

## Основные свойства системы

### Криптографическая целостность

* Подпись: **Ed25519**
* Хэширование payload: **SHA-256**
* Подписывается строго определённый набор полей

### Детерминированность

* Канонизация JSON по **RFC 8785 (JCS)**
* Один и тот же объект данных всегда даёт одинаковый байтовый поток
* Исключены неоднозначности сериализации

### Zero-trust модель

* Любой входящий конверт считается недоверенным
* Все проверки выполняются заново при верификации
* Отсутствуют «доверенные пути» внутри пайплайна

### Защита от replay-атак

* Используется комбинация `(key_id, nonce, issued_at)`
* Поддерживается stateful replay guard
* Архитектура допускает распределённые backends

## Архитектурный состав

### Nekhebet Core

Ядро системы, не зависящее от транспорта, БД или источника данных.

Содержит:

* канонические структуры данных,
* канонизацию,
* создание и верификацию конвертов,
* криптографические операции,
* систему политик,
* метрики и конфигурацию.

Core — **самодостаточный слой**, пригодный для встраивания в любые приложения.

---

### Хранилище (Store)

Опциональный компонент, реализующий долговременное хранение конвертов.

Текущая реализация — **гибридная**:

* PostgreSQL — индексы, метаданные, запросы
* LMDB — бинарные конверты, zero-copy доступ

Архитектура допускает альтернативные реализации:

* объектные хранилища,
* append-only журналы,
* специализированные time-series решения.

---

### Ingest-модули

Адаптеры, преобразующие внешние источники данных в SignedEnvelope.

Ingest не является обязательной частью системы и существует как:

* пример использования Core,
* точка расширения под конкретные источники.

## Типовые сценарии применения

### Аудит и журналирование

**Задача**
Создание неизменяемого журнала действий с возможностью независимой проверки.

**Как используется Nekhebet**

* Каждое действие оформляется как SignedEnvelope
* Конверты сохраняются в хранилище
* Проверка целостности возможна постфактум

**Результат**

* Доказуемая неизменяемость
* Отслеживаемое происхождение событий
* Пригодность для compliance и расследований

---

### Event-driven системы

**Задача**
Безопасный обмен событиями между сервисами.

**Как используется Nekhebet**

* События подписываются на стороне источника
* Верификация выполняется получателем
* Replay-атаки и подмена исключены

**Результат**

* Идемпотентность
* Проверяемый источник
* Чёткая типизация событий

---

### Сбор данных из внешних источников

**Задача**
Приём данных из недоверенных или полу-доверенных систем.

**Как используется Nekhebet**

* Данные упаковываются в SignedEnvelope
* Фиксируется источник и время создания
* Проверка выполняется при приёме и при хранении

**Результат**

* Гарантированная целостность
* Отслеживаемая цепочка происхождения

---

### Криптографически строгие пайплайны

**Задача**
Детерминированные данные для систем, где важна повторяемость (DLT, расчёты, архивы).

**Как используется Nekhebet**

* RFC 8785 гарантирует одинаковое представление данных
* Подпись не зависит от среды исполнения

**Результат**

* Воспроизводимость
* Возможность независимой проверки без контекста системы-источника

## Пример базового сценария

```python
# Создание и подпись события
unsigned = create_envelope(
    event_type="system.action",
    payload={"action": "update", "object": "config"},
    source="service-A",
    key_id="service-A-key"
)

signed = sign_envelope(unsigned, signing_context)

# Передача / хранение
data = serialize(signed)

# Приём и верификация
received = deserialize(data)
result = verify_envelope(received, replay_guard)

if result.valid:
    handle_event(received.payload)
```

## Charon Vessel (опциональный компонент)

**Charon Vessel** — это отдельный компонент, который может использоваться совместно с Nekhebet для управления физическим размещением и жизненным циклом данных.

### Роль Charon Vessel

* перенос и ротация файлов между устройствами,
* атомарные операции перемещения,
* cold-storage и архивирование.

### Типовой сценарий интеграции

* Активные конверты хранятся в LMDB / PostgreSQL
* Устаревшие события сериализуются в файлы
* Charon Vessel переносит их в архивное хранилище
* В индексе остаётся ссылка на архив

Charon Vessel **не является частью Nekhebet**, но логически хорошо дополняет его в системах с длительным хранением данных.
